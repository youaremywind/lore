故障排查从定位失败边界开始。

不要一开始就调召回阈值。多数问题来自设置、认证、过期 hooks、不完整模型设置或弱记忆结构。

## 按这个顺序检查

1. **服务器健康** — `/api/health` 有响应，Web Console 能打开。
2. **数据库和迁移** — `DATABASE_URL` 正确；PostgreSQL 有 `vector`；需要中文召回时有 `pg_jieba` 或 `zhparser`。
3. **设置** — `/setup` 显示 embedding、View LLM 和启动记忆设置完整。
4. **认证** — 受保护服务器能从 REST 和 MCP 客户端收到 `Authorization: Bearer <API_TOKEN>`。
5. **Agent 连接** — MCP URL 包含正确 `client_type`；插件文件和 hooks 已安装。
6. **运行时重载** — 配置、hook 或插件变更后 Agent 已重启。
7. **记忆质量** — path、disclosure、glossary 和 priority 能清楚描述未来 prompt。
8. **召回设置** — 上述检查通过后再调阈值。

## 卸载

从机器上完全移除 Lore：

```bash
# 停止并删除容器、网络和卷
docker compose down -v

# 删除本地 Lore 状态
rm -rf ~/.lore

# 移除 Claude Code Lore 集成
rm -f ~/.claude/lore-guidance.md
# 从 ~/.claude/CLAUDE.md 中删除 @~/.claude/lore-guidance.md 导入行

# 移除 Codex MCP 注册
codex mcp remove lore || true

# 从 ~/.openclaw/openclaw.json 中移除 OpenClaw 插件 entry

# 移除 Pi 扩展
rm -rf "${PI_CODING_AGENT_DIR:-$HOME/.pi/agent}/extensions/lore"

# 移除 Hermes 插件 symlink
rm -f ~/.hermes/skills/lore_memory
```

如果你在 shell profile 中添加过 `LORE_BASE_URL` 和 `LORE_API_TOKEN`，也一并移除。
